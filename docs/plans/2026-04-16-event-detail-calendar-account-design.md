# 详情所属账户展示设计

**日期：** 2026-04-16

**目标：** 在事件与提醒事项详情页的来源卡片中，同时展示日历/列表名称与所属账户/邮箱，避免不同账户下同名来源造成混淆。

## 背景

当前详情页的来源卡片只显示 `calendar.title`。这在单账户场景下基本可用，但在多账户场景中会出现明显歧义：

1. 不同邮箱下可能都存在名为“日历”的默认分类。
2. 用户虽然能在设置页看到这些来源按账户分组展示，但进入某条事件或提醒事项详情后，无法直接判断其属于哪个账户。
3. 对会议邀请、工作与个人日历并存的办公场景，这会降低确认与排错效率。

代码现状表明，账户层级信息并不缺失：

1. 事件详情页当前使用 `event.calendar.title` 渲染所属日历。
2. 提醒事项详情页当前使用 `reminder.calendar.title` 渲染所属列表。
3. 设置页已经按 `calendar.source?.title` 分组日历。
4. 因此这次问题主要是详情页信息架构不完整，而不是数据接入不足。

## 用户目标

1. 在详情页一眼区分“这是哪个账户下的哪个来源”。
2. 不显著增加详情页高度，不破坏当前紧凑布局。
3. 继续保持深色浮层卡片风格，与现有详情页一致。

## 非目标

1. 不修改设置页的数据源组织方式。
2. 不新增账户筛选或切换交互。
3. 不把事件详情重构为双栏或更重的信息面板。

## 方案对比

### 方案 A：单卡片双行展示

- 卡片标题保持“所属日历”。
- 第一行显示日历名。
- 第二行显示账户/邮箱，使用次级文字层级。

**优点：**

1. 改动最小，和现有布局最兼容。
2. 信息层级清晰，主次关系自然。
3. 只增加一行内容，不会明显拉长详情页。

**缺点：**

1. 长邮箱需要截断策略。

### 方案 B：拆成两张卡片

- 一张“所属账户”。
- 一张“所属日历”。

**优点：**

1. 语义最直白。

**缺点：**

1. 占用垂直空间更大。
2. 会让本就信息较多的详情页继续变长。

### 方案 C：合并为单行字符串

- 示例：`yangyi13@lenovo.com / 日历`

**优点：**

1. 最省空间。

**缺点：**

1. 主次关系不清晰。
2. 邮箱较长时更容易显得拥挤。

## 选定方案

采用方案 A：保留一张“所属日历”卡片，使用双行展示账户层级。

## UI/UX 设计

### 信息层级

事件详情中的“所属日历”和提醒事项详情中的“列表”卡片统一改为：

1. 卡片标题保持当前语义，不改命名
2. 主信息：日历/列表名称，例如 `日历`、`工作`
3. 次信息：账户来源，例如 `yangyi13@lenovo.com`

这样可以把用户真正关心的语义表达为：

`账户/邮箱 -> 日历/列表分类 -> 当前事项`

### 视觉规则

1. 保持现有图标与卡片容器，不新增额外图标。
2. 日历名继续使用现有正文主层级样式。
3. 账户/邮箱使用更小一档、次级灰色文字，避免和主信息竞争。
4. 长账户名默认单行，中间截断，尽量保留用户名与域名两端识别性。
5. 悬停时显示完整值，兼顾紧凑布局与完整信息可达性。

### 边界规则

1. 当 `calendar.source?.title` 为空时，回退为当前行为，只显示日历名。
2. 当账户名与日历名完全相同或仅大小写不同，隐藏第二行，避免机械重复。
3. 对 iCloud、Other 等非邮箱来源，也按同一规则展示其 source title，因为它们同样是账户层级语义。

## 实现设计

### 展示语义收敛

为避免把去重、空值和裁剪判断散落在视图里，本次新增一个轻量展示语义：

```swift
struct CalendarContextPresentation: Equatable {
    let calendarTitle: String
    let accountTitle: String?
}
```

职责如下：

1. 接收 `calendar.title` 与 `calendar.source?.title`。
2. 统一处理去首尾空格。
3. 统一决定是否显示第二行账户信息。

### 视图改动

1. `EventDetailWindowView` 继续使用现有 `EventDetailRow` 容器样式。
2. `ReminderDetailWindowView` 继续使用现有 `ReminderDetailRow` 容器样式。
3. 为两类 row 增加可选的次信息展示能力，而不是新增平行组件。
4. 展示语义收敛在 `CalendarContextPresentation` 与 `EKCalendar` 扩展中，避免事件和提醒事项各自拼装逻辑。

## 风险与处理

1. **长邮箱破坏卡片节奏**
   - 处理：次信息单行中间截断，并提供 hover 完整提示。

2. **source title 与日历名重复**
   - 处理：在展示语义层统一去重，避免详情页出现两行相同文本。

3. **部分来源不是邮箱地址**
   - 处理：仍展示 source title，因为它对用户来说依然是账户来源标识。

## 验证方式

1. 多账户下同名“日历”事件，详情页可同时看到日历名与账户名。
2. 多账户下同名提醒事项列表，详情页可同时看到列表名与账户名。
3. 单账户或 source title 为空场景，详情页不出现空白第二行。
4. source title 与来源名重复时，不显示重复副文案。
5. 构建通过，现有详情页其他区块不回退。
