# 日程卡片 Metadata 行布局修订设计

**日期：** 2026-04-11

**目标：** 修正日程卡片在显示右上角 metadata 时对标题宽度的挤压问题，让时间与 metadata 共用首行，而标题独立占满下一行宽度。

## 背景

上一版 `EventCardView` 将主内容放在左侧 `VStack`，metadata 放在最右侧。这种结构满足了“右上角显示会议 / 重复提醒语义”的需求，但也带来了一个明显问题：

1. metadata 会直接压缩整个左侧内容列。
2. 标题因此提前换行。
3. 标题下方右侧出现大块空白，信息密度下降。

从用户截图看，这个问题在长标题的会议和提醒事项中都很明显。

## 用户确认

用户确认新的排版目标为：

1. 右上角内容与第一排时间保持在同一行。
2. 标题独立成行，并占满整个宽度。
3. 所有没有 metadata 的卡片也统一使用该结构，保证列表节奏一致。

## 方案对比

### 方案 A：三行结构重排

- 第一行：时间 + metadata
- 第二行：标题
- 第三行：副标题

**优点：**

- 直接满足用户预期
- 标题拿回完整宽度
- 结构简单，SwiftUI 布局稳定

**缺点：**

- 所有卡片高度会更统一，不再保留旧版部分卡片的“更扁平”感

### 方案 B：绝对定位 metadata 到右上角

- metadata 通过 overlay 固定在右上角
- 内容区域仍保留原 `VStack`

**优点：**

- 表面改动少

**缺点：**

- 容易与首行内容重叠
- 对动态文字和不同内容长度不稳

### 方案 C：仅对有 metadata 的卡片重排

- 有 metadata 的卡片使用三行结构
- 无 metadata 的卡片保持旧结构

**优点：**

- 改动更保守

**缺点：**

- 列表节奏不一致
- 视觉跳动更明显

## 选定方案

采用方案 A：所有卡片统一改为三行结构。

## UI/UX 设计

### 信息结构

卡片主体调整为：

1. `headerRow`
   - 左侧：时间
   - 右侧：metadata（如果存在）
2. `titleRow`
   - 标题独立成行
   - 继续支持最多两行
3. `secondaryRow`
   - 地点 / 日历名 / 提醒来源等次级信息

### 视觉规则

1. 首行使用较轻的次级文字层级，保持紧凑。
2. 标题继续是卡片的主要视觉焦点。
3. metadata 只参与首行宽度分配，不再影响标题和副标题的宽度。
4. 没有 metadata 的卡片仍保留同一首行结构，只是右侧为空。

### 交互规则

1. 卡片点击行为保持不变。
2. metadata 仍然只是展示信息，不增加单独点击热区。
3. Reminder 勾选按钮、进行中色条、取消态样式继续沿用原行为。

## 架构设计

### 视图调整

本次只需要修改 `EventCardView` 的布局组织，不改动 `CalendarItem` 的 metadata 派生语义：

1. 从单一内容 `VStack + trailing metadata` 改为卡片内容 `VStack`。
2. 将时间提取到首行 `HStack`。
3. 标题和副标题分别成为独立的下级行。

### 语义复用

会议 metadata、提醒重复 metadata 的来源保持不变，继续复用当前 `CalendarItem` 提供的语义属性，避免这次布局修订反向引入逻辑分叉。

## 非目标

1. 不改变 metadata 的业务逻辑。
2. 不调整 Teams 图标或提醒重复文案的具体语义。
3. 不改动时间轴、详情窗口或提醒详情页。

## 风险与处理

1. **统一三行结构后卡片高度略有变化**
   - 处理：维持现有字体和垂直间距级别，只调整行组织，不额外加大 padding。

2. **首行 metadata 过长挤压时间文本**
   - 处理：继续保持 metadata 紧凑单行显示，标题宽度不再受影响。

3. **没有 metadata 的卡片看起来过于稀疏**
   - 处理：统一使用相同的行间距和副标题策略，避免视觉断层。

## 验证方式

1. 带 metadata 的会议卡片：时间与 metadata 同行，标题占满下一行宽度。
2. 带 metadata 的提醒卡片：时间与重复规则同行，标题不再被挤压。
3. 无 metadata 的普通事件卡片：仍采用相同三行结构，视觉节奏统一。
4. 构建通过，不回退现有 metadata 语义和点击行为。
