# 设置窗口改版设计

**日期：** 2026-03-30

**目标：** 重构 Calendar Pro 的设置窗口视觉层级与导航交互，解决当前布局松散、缺少信息层级，以及侧边栏点击后明细区不切换的问题。

## 现状问题

1. 左侧导航仅使用 `NavigationSplitView + List(selection:)` 的默认外观，缺少明确的品牌头部、选中反馈和导航说明。
2. 右侧详情区直接渲染各个设置页内容，没有统一的页面标题、摘要说明和内容容器，导致视觉重心不稳定。
3. `SettingsRootView` 对导航状态的表达过于薄弱，实际交互依赖系统默认选中行为，导致侧边栏点击反馈不稳定。
4. 当前窗口尺寸与内容边距偏紧，信息虽然不多，但看起来仍显得拥挤和失衡。

## 设计目标

1. 保留 macOS 原生设置窗口气质，不做花哨装饰。
2. 左侧导航需要有稳定、明确的点击反馈，并由显式状态驱动右侧页面切换。
3. 右侧页面统一为“标题 + 说明 + 状态摘要/设置内容”的结构。
4. 保持现有设置功能与数据模型不变，优先调整容器、层级和交互实现。

## 方案比较

### 方案 A：自定义双栏设置壳层

- 左侧使用自定义导航列表按钮，而不是 `List(selection:)`
- 右侧统一增加标题区、说明文案和卡片式内容容器
- 使用显式 `selectedItem` 状态驱动切换

**优点：**
- 交互稳定，可完全控制选中态
- 视觉层级最好，便于后续继续扩展设置项
- 不依赖 `NavigationSplitView` 在 macOS 上的默认行为

**缺点：**
- 比局部修补多一些视图代码

### 方案 B：保留 `NavigationSplitView`，补齐样式与标签

- 保留现在结构
- 增加 `tag`、统一标题栏和容器样式

**优点：**
- 改动较小

**缺点：**
- 视觉改善有限
- 仍受默认导航组件行为影响

### 方案 C：回退 `TabView`

- 回到原来的分页标签结构

**优点：**
- 最省实现成本

**缺点：**
- 信息架构退化，不适合当前设置分区

## 选定方案

采用方案 A。

## 信息架构

### 左侧导航

- 顶部增加应用名和一句功能说明
- 中间为 4 个导航项：通用、菜单栏、日程、地区与节假日
- 每个导航项包含图标、标题、简短说明
- 当前选中项使用带浅色填充的圆角背景突出显示

### 右侧详情区

- 顶部统一展示当前页面标题与一句解释
- 页面主体使用统一的滚动区域和卡片分组
- “通用”页保留现有状态信息，但改成更可读的状态摘要卡片

## 交互设计

1. 点击左侧任一导航项，立即更新 `selectedItem`
2. 右侧根据 `selectedItem` 切换对应页面
3. 保留设置窗口打开时默认进入“通用”
4. `onAppear` 仍负责触发日历权限状态与日历列表刷新

## 实现边界

- 不修改 `SettingsStore`、`EventService` 的行为
- 不引入新的持久化字段
- 不调整已有设置项的业务逻辑

## 验证方式

1. 手动打开设置窗口，确认四个导航项均可切换
2. 检查不同页面的标题、说明和卡片样式是否统一
3. 构建项目并运行现有测试，确认没有引入编译回归
