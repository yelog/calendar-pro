# 日程详情侧边面板设计

## 背景

当前 Popover 只支持“选中日期后在下方显示日程列表”。用户可以看到当天有哪些日程，但无法在不跳出当前界面的情况下查看单个日程的详细信息。

本次需求是在下拉菜单中点击某个日程时，在界面左侧展示一个独立感更强的详情面板，用于承载该日程的详细信息。

## 需求分析

### 用户目标

- 在当前下拉菜单内快速查看某条日程的详细信息
- 不打断原有“看月历 + 切换日期 + 浏览当天日程”的流程
- 详情与列表同时可见，便于在多条日程之间快速切换

### 约束

- 当前应用基于 `NSPopover + SwiftUI` 实现，没有现成的第二窗口或副面板控制器
- 现有状态流以 `RootPopoverView -> CalendarPopoverView -> EventListView` 为主，事件详情应复用已有 `EKEvent` 数据
- 项目工作区当前已有未提交改动，应尽量减少对工程配置和非目标文件的扰动

### 成功标准

- 点击日程卡片后，Popover 左侧出现详情面板
- 切换到另一条日程时，详情内容同步更新
- 切换日期、无日程、权限关闭或筛选变化导致目标日程失效时，详情面板自动关闭
- 右侧原有日历与日程列表不被破坏

## 方案对比

### 方案 A：同一 Popover 内做可展开双栏

- 做法：Popover 根据是否选中日程，在原有主面板左侧插入一个详情面板，并整体增宽
- 优点：实现成本低，交互连续，关闭逻辑和状态管理简单
- 缺点：需要处理 Popover 宽度随内容变化

### 方案 B：新增独立 `NSPanel` 作为左侧详情窗口

- 做法：点击日程时创建一个跟随主 Popover 的辅助窗口，并手动定位在左侧
- 优点：视觉上更像“另一个窗口”
- 缺点：需要处理窗口定位、关闭同步、焦点切换、屏幕边界和多显示器问题，超出当前需求必要复杂度

### 方案 C：在当前主面板内替换底部列表为详情页

- 做法：点击日程后，用详情页替换原有日程列表区域
- 优点：不用处理宽度变化
- 缺点：不能同时看列表和详情，不符合“左边另一个面板”的表达

## 推荐方案

采用方案 A。

它最贴合当前架构，也最接近用户对“左边出现另一个面板”的感知。虽然本质仍是单个 Popover，但通过左右双栏布局、独立背景层次、竖向分隔线和固定面板宽度，可以形成清晰的“副面板”体验，同时避免引入第二个原生窗口的高维护成本。

## UI/UX 设计

## 整体布局

- 默认状态：保留现有单栏 Popover，宽度约 `340`
- 点击日程后：Popover 扩展为左右双栏，总宽度约 `620-660`
- 左栏：日程详情面板
- 右栏：保留当前月份导航、日历网格、日程列表和底部操作栏

## 左侧详情面板结构

- 顶部：`日程详情` 标题 + 关闭按钮
- 概览区：日历颜色标识、标题、日期/时间摘要
- 元信息区：展示日历名称、地点
- 备注区：展示 `notes`
- 空状态：若无地点和备注，展示“暂无更多详情”

## 视觉层次

- 详情面板使用独立圆角容器和更明显的背景层
- 面板顶部使用与日历颜色呼应的轻量强调色
- 当前选中的日程卡片使用高于普通卡片的描边和背景高亮
- 左右栏之间使用细竖线，强化“主面板 / 详情面板”分区

## 交互规则

- 点击日程卡片：打开或更新左侧详情面板
- 点击关闭按钮：关闭详情面板，但保留当前日期与日程列表
- 切换日期：清空当前选中日程，等待用户重新点击
- 日程刷新后若已选中项不存在：自动关闭详情面板

## 状态设计

- `selectedDate`: 当前选中的日期
- `selectedEventIdentifier`: 当前选中的日程 ID
- `eventsForSelectedDate`: 当前日期对应的日程列表

其中 `selectedEventIdentifier` 只用于详情面板和列表高亮，不改变事件加载逻辑。

## 测试策略

- 单元测试补充 `CalendarPopoverViewModel` 的“日程选择/清除”行为
- 验证切换日期时会清空当前选中日程
- 验证显式关闭详情时会清空当前选中日程

## 实施范围

- 修改 `CalendarPopoverViewModel` 管理选中日程状态
- 修改 `RootPopoverView` 协调日期切换、事件刷新和详情面板失效逻辑
- 修改 `CalendarPopoverView` 实现左右双栏布局
- 修改 `EventListView` / `EventCardView` 支持点击与高亮
