# 日程详情仿下拉悬浮窗设计

## 背景

第一版独立详情窗已经解决了“点击日程没反应”的功能问题，但从实际效果看仍有两个明显偏差：

- 详情窗尺寸过大，尤其在备注或会议邀请很长时，会变成一个桌面级大窗体
- 详情窗的边框、标题栏和内部层级与当前下拉菜单不一致，视觉上不像“从下拉菜单呼出来的悬浮框”

用户当前确认的新目标不是回退到内嵌详情面板，而是保留“独立 macOS 窗口”的技术路线，同时把它收敛成一个更像下拉菜单附属浮层的轻量详情窗。

## 需求分析

### 用户目标

- 点击下拉中的某条日程后，立即看到完整详情
- 详情仍然是独立窗口，不挤压或改写当前下拉布局
- 详情窗在视觉上与当前下拉菜单属于同一套界面语言
- 长备注、会议邀请、长链接不再把窗口撑成一个大面板

### 非目标

- 不改成 Popover 内嵌双栏
- 不引入第二个 `NSPopover` 与状态栏主 Popover 争生命周期
- 不把详情窗升级为标准主窗口或设置窗口样式

### 约束

- 应用主壳仍是 `NSStatusItem + NSPopover + SwiftUI`
- 日程列表和详情选中状态仍然由 `RootPopoverView` 侧驱动
- 列表中存在 `.event` 和 `.reminder` 两类数据，只有 `.event` 可以打开详情
- 当前已经有一版 `NSPanel` 基础实现，后续应优先在这条链路上收敛样式，而不是推翻重写
- 工程文件由 [`tools/generate_xcodeproj.rb`](/Users/yelog/workspace/swift/calendar-pro/tools/generate_xcodeproj.rb) 生成，新增文件后必须重生成

### 成功标准

- 详情窗固定宽度，与日历下拉宽度一致，目标值为 `340pt`
- 详情窗高度不再随内容无限增长，而是在一个明确范围内夹取
- 长内容进入局部滚动区，不再驱动整个窗口变大
- 详情窗去掉“标准桌面窗口”的强存在感，视觉更像附属浮层
- 点击另一条日程时复用同一个浮层窗口并就地更新内容
- 关闭 Popover、切换日期、事件失效或再次点击同一条日程时，详情窗按预期关闭

## 方案对比

### 方案 A：继续使用当前标准 `NSPanel`，只缩小尺寸

- 做法：保留现有标题栏样式，只把宽高调小
- 优点：改动最少
- 缺点：只能缓解尺寸问题，无法解决“像另一个系统窗口”的视觉割裂

### 方案 B：使用轻量化 `NSPanel`，视觉上仿 Popover 悬浮卡片

- 做法：保留独立窗口，但改为无明显标题栏的轻量浮层风格，固定宽度、限制最大高度、内容局部滚动
- 优点：同时满足“独立窗口”和“像从下拉呼出”的要求，是最平衡的方案
- 缺点：需要统一样式 token，并重新梳理详情视图布局

### 方案 C：改成第二个 `NSPopover`

- 做法：点击日程后再打开一个次级 Popover
- 优点：视觉最接近系统菜单的子气泡
- 缺点：生命周期与主 Popover 强耦合，不再符合“独立窗口”的原始要求，交互也更脆弱

## 推荐方案

采用方案 B：保留独立 `NSPanel`，但将其重构为仿下拉菜单悬浮卡片的轻量浮层窗口。

这条路线不改变现有功能架构，只收敛窗口样式、尺寸策略和内容布局。它既满足用户对独立窗口的要求，也避免落回双栏 Popover 或第二个 Popover 的复杂交互。

## 架构设计

### 组件职责

- `RootPopoverView`
  - 继续管理 `selectedDate`、`selectedEventIdentifier` 和当天列表
  - 负责发出“打开详情窗 / 关闭详情窗”的意图
  - 在日期切换、权限变化、事件失效时统一收口

- `CalendarPopoverView`
  - 继续保持单栏菜单布局
  - 提供与详情窗共享的视觉宽度和背景风格基线

- `EventListView` / `EventCardView`
  - 只负责把事件点击往上抛
  - 通过选中态和箭头表达“该条目会呼出详情浮层”

- `PopoverController`
  - 作为 SwiftUI 与 AppKit 的桥梁
  - 持有单实例详情 presenter
  - 在 Popover 关闭时同步关闭详情浮层

- `EventDetailWindowController`
  - 继续使用单实例 `NSPanel`
  - 负责固定宽度、限制高度、屏幕边界定位和内容更新
  - 从“标准面板窗口”收敛为“轻量悬浮浮层”

- `EventDetailWindowView`
  - 负责详情信息的信息架构和视觉层级
  - 把头部、概览区与详情正文拆成固定区和滚动区

### 数据流

1. 用户点击 Popover 中某条 `.event`
2. `RootPopoverView` 基于 `selectedEventIdentifier` 判断是打开、更新还是关闭
3. `PopoverController` 接收意图并调用 `EventDetailWindowController`
4. `EventDetailWindowController` 复用现有面板，更新内容与位置
5. 窗口主动关闭时，通过回调反向清理 SwiftUI 侧选中态

### 状态规则

- `selectedEventIdentifier` 只表示“当前详情浮层对应的事件”
- 同一事件再次点击表示切换为关闭态
- 选中事件从列表中消失时，详情浮层必须自动关闭
- 详情浮层关闭后，列表高亮必须同步清空

## UI/UE 设计

### 视觉定位

详情窗应该被感知为“当前下拉菜单的延展层”，而不是“应用新开了一个完整窗口”。

因此视觉上应遵循三条原则：

- 与下拉菜单共享相同宽度基线和背景语气
- 去掉强窗口语义，只保留必要的关闭与阴影
- 让内容信息密度更接近菜单卡片，而不是文档型详情页

### 尺寸策略

- 固定宽度：`340pt`
- 最小高度：`280pt`
- 理想高度：`360pt`
- 最大高度：`440pt`

规则如下：

- 宽度始终固定，不因标题、链接或备注变长而变宽
- 高度在最小值和最大值之间夹取
- 内容超过最大高度后，窗口整体不继续放大，而是进入内部滚动

### 内容布局

建议采用固定头部 + 可滚动正文的结构：

1. 头部
   - 小标签“日程详情”
   - 日历颜色点 + 事件标题
   - 右上角关闭按钮

2. 概览卡
   - 日期
   - 时间区间

3. 正文滚动区
   - 所属日历
   - 地点
   - 链接
   - 备注

4. 空状态
   - 若没有地点、链接、备注，则显示“暂无更多详情”

### 视觉 token

- 与当前 Popover 共用 `340pt` 宽度
- 背景继续使用浅层渐变和低对比色块，而不是大面积深层容器叠加
- 外层圆角建议 `14-16pt`
- 内部 section 间距控制在 `8-12pt`
- 标题字号从当前偏重的 `20pt` 下调至 `15-16pt`
- 元信息标签使用 `11-12pt`，保持菜单级轻量层次

### 长内容处理

当前截图中最大的问题来自长备注和会议邀请文本。新的滚动策略应为：

- 头部和概览区固定显示，不滚动
- 正文区单独包裹在 `ScrollView` 中
- 备注文本允许多行，但只在正文滚动区里扩展
- 长链接默认两行截断，必要时支持文字选择复制

这样既保留完整信息，又不会让详情窗变成一个大文档窗口。

### 位置与动效

- 默认显示在 Popover 左侧
- 水平间距 `8-10pt`
- 优先与事件列表区域顶部对齐；若当前实现难以稳定拿到行级坐标，则退回到与 Popover 顶部对齐
- 左侧空间不足时回退到右侧
- Y 轴始终夹取到屏幕可见范围内
- 若实现成本可控，可增加 `150-180ms` 的轻微淡入位移动画；若会引入额外复杂度，则动画不是硬要求

## 交互规则

- 点击未选中的日历事件：打开详情浮层
- 点击另一条日历事件：复用同一浮层并刷新内容
- 点击当前已选事件：关闭详情浮层
- 点击提醒事项：不打开详情
- 切换日期、筛选、权限或事件失效：关闭详情浮层
- 关闭 Popover：详情浮层同步关闭

## 风险与边界

- 若继续使用标准标题栏 `NSPanel`，即使尺寸变小，视觉仍会偏离目标
- 若让整个详情视图参与 `fittingSize` 自适应，长备注会再次把窗口撑大
- 若长链接不做截断或局部滚动，会破坏固定宽度策略
- 若后续尝试改成第二个 `NSPopover`，会带来新的生命周期和点击穿透问题

## 测试策略

- 单元测试：新增尺寸计算纯函数测试，覆盖固定宽度、高度最小值、最大值夹取
- 单元测试：保留并补充窗口定位测试，验证左右回退与 Y 轴夹取
- 单元测试：保留 `PopoverController` presenter 联动测试，确保显示与关闭语义不回退
- 手动验证：
  - 长备注不会把窗口整体撑大
  - 宽度与下拉菜单一致
  - 点击不同事件时为同一浮层更新内容
  - 关闭 Popover 时详情浮层同步消失
