# 已取消日程样式设计

**日期：** 2026-04-10

**目标：** 在弹层日程列表和详情窗口中，为 EventKit 标记为已取消的日程提供清晰但克制的视觉语义，避免与普通过去事件混淆。

## 背景

当前日程列表和详情窗口都会直接按普通事件渲染 `EKEvent`。即使事件在外部日历源中已经被取消，界面上也不会给出区别化表现。

这会带来两个问题：

1. 用户无法一眼识别该日程已经失效。
2. 已取消事件在深色背景下容易与普通过去事件混在一起。

## 用户确认

本次调整范围包括：

1. 下拉弹层中的日程时间轴卡片。
2. 日程详情窗口。

用户确认采用接近 Apple Calendar 的方向：

1. 删除线表达“取消”。
2. 整体视觉弱化，但仍保留原事件上下文。
3. 详情窗口额外显示轻量状态标签。

## 方案对比

### 方案 A：仅降低透明度并给标题加删除线

- 列表和详情只做最小样式处理
- 不新增额外状态标签

**优点：**
- 改动最小
- 风险低

**缺点：**
- 在深色界面下辨识度有限
- 容易与“普通过去事件”混淆

### 方案 B：删除线 + 弱化卡片 + 轻量状态标签

- 列表标题加删除线
- 卡片整体改为弱化语义，不再沿用进行中高亮
- 详情窗口补充 `已取消` 状态胶囊

**优点：**
- 语义清楚
- 与原生 Calendar 风格接近
- 对现有布局侵入较小

**缺点：**
- 需要补充事件状态语义位和若干样式分支

### 方案 C：使用红色强调取消态

- 通过红色背景、红色边框或红色标签强调取消

**优点：**
- 最醒目

**缺点：**
- 视觉侵略性较强
- 容易与错误态、冲突态混淆

## 选定方案

采用方案 B：删除线 + 弱化卡片 + 轻量状态标签。

## 架构设计

### 语义来源

新增一条轻量事件状态语义链路：

1. `CalendarItem` 为 `EKEvent` 暴露“是否已取消”的只读语义。
2. `EventCardView` 基于该语义决定卡片文本、背景、边框和颜色点样式。
3. `EventDetailWindowView` 基于同一语义渲染标题删除线与状态胶囊。

`EKReminder` 不参与该语义判断，维持现有行为。

### 视觉规则

#### 列表卡片

1. 标题使用删除线。
2. 时间和副标题统一降为次级语义。
3. 卡片背景和边框采用中性弱化样式。
4. 左侧日历颜色点保留，但降低存在感，避免丢失来源信息。
5. 如果事件被选中，仍保留选中态边框，但优先体现“已取消”。

#### 详情窗口

1. 标题使用删除线。
2. 标题区域增加轻量 `已取消` 状态胶囊。
3. 日期、时间、地点、参会人、备注等详情继续完整展示，方便核对原始上下文。

### 优先级规则

1. 已取消语义高于 `past`、`ongoing`、`selected` 的卡片视觉分支。
2. 当前时间线 marker 继续按时间位置展示，不因取消态而消失。
3. 全天取消事件沿用同一取消态语义，但不调整其所在分区布局。

## 非目标

1. 不解析标题中的“已取消”“Canceled”等文本来猜测状态。
2. 不改动提醒事项的视觉语义。
3. 不改动日程的排序、分组、时间轴定位和详情内容结构。

## 风险与处理

1. **部分外部日历源没有写入 EventKit 取消状态**
   - 处理：只信任 EventKit 原生状态；没有状态就按普通事件显示，避免误判。

2. **取消态与进行中高亮冲突**
   - 处理：取消态优先，取消事件不再使用进行中高亮卡片。

3. **选中态过强导致取消语义被冲淡**
   - 处理：保留选中边框，但整体继续使用弱化视觉，不恢复普通选中底色。

## 验证方式

1. 时间轴列表中的取消事件能与普通事件、过去事件明显区分。
2. 正在当前时间段内的取消事件不会显示为进行中高亮卡片。
3. 详情窗口中标题带删除线，且可见 `已取消` 状态标签。
4. 普通事件、提醒事项、已完成提醒的现有样式不回归。
