# 日程参与状态展示与回复设计

**日期：** 2026-04-11

**目标：** 在弹层日程列表和日程详情页中展示当前用户的参与状态，并在详情页内直接完成 Accept / Maybe / Decline 回复，重复日程支持在“仅本次”和“整个系列”之间选择作用范围。

## 背景

当前代码已经具备两块基础能力：

1. `EventCardView` 右上角已有 metadata 展示位。
2. `EventDetailWindowView` 已能展示 `event.attendees` 及各参会人的 `participantStatus`。

缺口在于：

1. 还没有突出“当前用户自己的参与状态”。
2. 详情页里不能直接回复邀请。
3. 列表右上角也没有用参与状态替代会议 metadata。

## 用户确认

1. 列表右上角在未回复时不显示状态。
2. 对重复日程，用户点击状态后需要可选“仅本次”或“应用到整个系列”。
3. 允许采用 EventKit 运行时可用但未在公开 Swift 头文件中暴露的参与状态写入能力，以换取原生式的应用内直接回复体验。

## 方案对比

### 方案 A：只展示状态，回复时跳系统日历

- 优点：完全依赖公开 API，最稳。
- 缺点：不能满足“在详情页内直接选择”的核心需求。

### 方案 B：应用内直接回复，失败时回退系统日历

- 优点：最接近 Apple Calendar 的交互路径。
- 优点：可以复用现有详情窗、列表刷新链路和 metadata 布局。
- 缺点：状态写入依赖 EventKit 运行时属性，存在不同日历源兼容性风险。

### 方案 C：通过 AppleScript 驱动系统日历回复

- 优点：理论上可借系统完成修改。
- 缺点：脆弱、权限复杂、可维护性差。

## 选定方案

采用方案 B。

## UI/UX 设计

### 列表卡片

1. 列表右上角改为组合 metadata，而不是单一 metadata。
2. 当前用户参与状态与会议平台图标、参会人数需要并存显示。
3. 为控制宽度，列表中的参与状态采用 icon-only 方案，只显示图标与语义颜色，不显示 `Accepted`、`Maybe`、`Declined` 文案。
4. 若当前用户尚未回复，则不显示参与状态 token。
5. 若事件没有会议 metadata，则仅显示参与状态 token；若事件也没有参与状态，则继续沿用现有会议 / 重复提醒 metadata 规则。

### 详情页头部

1. 在标题区域增加当前用户参与状态徽标。
2. 已回复时直接显示当前状态，作为详情页的顶部摘要之一。

### 详情页回复区

1. 新增 `Response` 区块。
2. 区块内提供三态选项：`Accept`、`Maybe`、`Decline`。
3. 当前选项高亮，未选中项用次级描边按钮展示。
4. 若当前事件不支持在本应用内修改参与状态，则只展示已存在状态，不展示交互按钮。
5. 对尚未回复的邀请，即使事件没有显式暴露“当前用户 attendee”，只要存在参会人列表且当前用户不是 organizer，也应展示 `Response` 区块与三态选项。

### 重复日程交互

1. 对属于重复系列的邀请，点击三态按钮后弹出作用范围确认。
2. 选项为：`Only This Event`、`Entire Series`、`Cancel`。
3. 代码层面使用 `EKSpan.thisEvent` / `EKSpan.futureEvents` 执行保存；UI 对外文案使用“整个系列”以贴合用户需求。

## 架构设计

### 共享语义层

在 `CalendarItem.swift` 中补充一个共享的参与状态语义层：

1. `EventParticipationChoice` 统一建模 `Accept` / `Maybe` / `Decline`。
2. `EKEvent` 扩展负责：
   - 读取当前用户参与状态
   - 判断当前事件是否具备回复上下文
   - 判断是否允许修改参与状态
   - 通过运行时 `participationStatus` 完成写入，并调用 `saveEvent(_:span:)` 保存

其中“具备回复上下文”的判断需要兼容一种常见邀请场景：

1. 事件有大量 `attendees`
2. 当前用户并非 organizer
3. 但底层并没有把当前用户显式放进 `attendees.first(where: \ .isCurrentUser)`

在这种情况下，详情页仍应展示参会状态区块，而不是因为 `pending` / `nil` 被隐藏。

### 视图层

1. `EventCardView` 右上角 metadata 改为 token 列表，允许参与状态与会议 metadata 同时显示。
2. `EventDetailWindowView` 增加参与状态摘要与回复区。
3. 详情页在本地维护一个显示状态，以确保保存后无需等待外部刷新即可立即更新 UI。

### 刷新链路

保存成功后沿用现有 `EKEventStoreChanged -> EventService.storeChangeRevision -> RootPopoverView.refreshEventsForCurrentSelection()` 链路，刷新列表与选中事件数据。

## 风险与处理

1. **运行时写入能力并非公开 Swift API**
   - 处理：所有 KVC 访问都先经过运行时可用性检查；失败时回退为错误提示，不做静默失败。

2. **不同日历源对参与状态修改支持不同**
   - 处理：仅在存在当前用户参与上下文且源允许修改时展示交互按钮。

3. **重复日程“整个系列”在底层依赖 `futureEvents`**
   - 处理：第一版先采用 EventKit 可用 span；若后续验证发现个别源语义不一致，再单独补范围策略修正。

## 非目标

1. 不重做参会人列表样式。
2. 不在列表卡片上直接提供可点击的快速回复按钮。
3. 不引入 AppleScript 或额外系统自动化能力。

## 验证方式

1. 邀请事件已接受时，列表右上角显示绿色图标态参与状态 token。
2. 会议邀请已接受时，列表右上角同时显示参与状态图标与 Teams 图标/人数。
3. 邀请事件未回复时，列表右上角不显示参与状态。
4. 详情页中能看到当前状态，并能切换到 Accept / Maybe / Decline。
5. 重复日程点击状态后，会弹出“仅本次 / 整个系列”选择。
6. 状态保存成功后，详情页立即更新，列表在 store change 后同步更新。
