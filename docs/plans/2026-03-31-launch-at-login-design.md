# 开机启动设置设计

## 背景

Calendar Pro 目前已经具备完整的设置窗口，但缺少“登录后自动启动应用”的能力。作为菜单栏应用，这属于高频的基础偏好项，缺失后会增加用户每次登录后的手动启动成本。

## 需求分析

### 用户需求
- 用户希望在设置中直接开启或关闭“开机启动”。
- 用户期望该开关修改后立即生效，不需要重启应用。
- 当系统拒绝或暂时无法完成注册时，用户需要看到明确反馈，而不是误以为设置已经成功。

### 成功标准
- 设置窗口中可见且可理解的“开机启动”选项。
- 开关状态能够正确反映系统真实注册状态。
- 开启或关闭后立即调用系统登录项注册能力。
- 注册失败时显示错误信息，并让界面恢复到真实状态。

## 方案比较

### 方案 A：使用 `SMAppService.mainApp` 直接注册主应用

- 在设置层增加“开机启动”布尔状态
- 通过 `ServiceManagement` 注册/取消注册当前主应用
- 在 UI 中展示当前状态和失败原因

**优点：**
- 符合 macOS 14+ 原生能力
- 不需要额外 helper target
- 实现简单，维护成本最低

**缺点：**
- 需要处理系统状态与 UI 状态同步

### 方案 B：仅持久化本地偏好，不接系统注册

**优点：**
- 实现最简单

**缺点：**
- 不能真正实现开机启动
- 容易制造“看起来开启、实际上无效”的错误体验

### 方案 C：引入 legacy login helper

**优点：**
- 兼容旧系统

**缺点：**
- 当前工程最低版本为 macOS 14，明显过度设计
- 需要新增 target 和更多分发配置

## 选定方案

采用方案 A。

## UI/UE 设计

### 信息架构
- 入口放在“通用”页，不新增新的设置分区。
- 在“当前状态”摘要之外，增加独立的“启动行为”分组。

### 交互设计
- 使用单个 `Toggle` 表达“登录后自动启动 Calendar Pro”。
- 在开关下方增加一句辅助说明，明确语义是“当前登录用户登录后自动启动”。
- 开关切换时立即提交系统注册操作。
- 若操作失败，在当前分组内展示错误提示文案。

### 视觉设计
- 继续沿用现有 `GroupBox` + 简短说明的原生设置风格。
- 不引入额外弹窗，避免中断操作流。
- 错误提示使用系统红色语义色，保持可读性与克制感。

## 架构设计

### 状态归属
- “开机启动”不并入 `MenuBarPreferences`。
- 在 `SettingsStore` 中新增单独的 `launchAtLoginEnabled`、`launchAtLoginStatusMessage` 发布状态。

### 系统能力抽象
- 新增 `LaunchAtLoginControlling` 协议，隔离系统 API 与业务状态。
- 生产实现使用 `SMAppService.mainApp`。
- 测试使用 fake controller，覆盖成功与失败分支。

### 数据流
1. `SettingsStore` 初始化时读取系统登录项状态。
2. `GeneralSettingsView` 通过 binding 展示开关。
3. 用户切换开关时，视图调用 `SettingsStore.setLaunchAtLoginEnabled(_:)`。
4. `SettingsStore` 通过 controller 执行注册或反注册。
5. 成功时刷新发布状态；失败时记录错误文案并回滚开关展示。

## 错误处理

- `SMAppService` 调用抛错时，向 UI 暴露可读错误信息。
- 对于“需要用户批准”或“未找到”这类状态，统一映射为可理解描述。
- 不因为注册失败影响其他设置页或菜单栏主流程。

## 测试策略

1. `SettingsStore` 默认会读取 controller 当前状态。
2. 开启成功后，发布状态为开启且无错误。
3. 关闭成功后，发布状态为关闭且无错误。
4. 开启失败后，状态回滚且包含错误提示。
5. 关闭失败后，状态回滚且包含错误提示。

## 实现边界

- 不新增 helper app。
- 不调整现有菜单栏展示逻辑。
- 不修改应用启动主流程，只补充设置与系统注册能力。
