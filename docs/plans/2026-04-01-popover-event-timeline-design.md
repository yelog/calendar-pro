# 下拉日程时间线设计

**日期：** 2026-04-01

**目标：** 在 Calendar Pro 的下拉日程区域中引入紧凑型 timeline，让用户快速判断当前时间落在当天哪一段行程中，并同时兼容日历事件与提醒事项。

## 背景

当前下拉面板已经具备：

- 选中日期后显示当天事件/提醒事项列表
- 按开始时间分组
- 当天自动滚动到进行中事件或下一个时间组
- 点击事件打开独立详情浮层，点击提醒打开提醒详情

但当前展示仍是离散卡片列表，缺少明确的时间轴语义。用户需要自己从时间文本推断“现在处在哪个日程里”，效率偏低。

## 需求分析

### 用户目标

1. 在下拉中直观看到当天行程的时间推进关系。
2. 用一条当前时间标记快速定位“现在在哪个日程附近”。
3. 提醒事项也要纳入时间线，而不是被排除到时间线外。
4. 保持现有点击交互：事件打开详情，提醒打开提醒详情。

### 非目标

1. 不把当前 340pt 的下拉改造成完整日视图 scheduler。
2. 不增加新的设置项或持久化字段。
3. 不改变 EventKit 拉取和详情浮层的总体架构。

### 约束

1. Popover 宽度固定为 [`PopoverSurfaceMetrics.width`](/Users/yelog/workspace/swift/calendar-pro/CalendarPro/Views/Popover/PopoverSurfaceMetrics.swift) 的 `340pt`。
2. 日程区域当前高度约束为 `200pt`，信息密度必须受控。
3. 提醒事项存在两类情况：
   - 有明确到期时间
   - 只有日期、没有明确时分
4. 当前代码中 [`CalendarItem.startDate`](/Users/yelog/workspace/swift/calendar-pro/CalendarPro/Features/Events/CalendarItem.swift) 对提醒事项直接使用 `dueDateComponents` 转成 `Date`，若不额外区分，只有日期的提醒会被误判为 `00:00`。

## 方案对比

### 方案 A：紧凑型垂直时间线列表

- 左侧是时间线和时间标签
- 右侧是对应时间组的卡片
- 当前时间线作为“现在”标记插入时间轴

**优点：**
- 最适合现有 340pt 宽下拉
- 可以复用当前按时间分组和自动滚动逻辑
- 视觉与现有卡片列表衔接自然

**缺点：**
- 不是严格按时长比例绘制的完整日程表

### 方案 B：完整日视图 scheduler

- 按小时切片，项目按时间跨度占据纵向空间

**优点：**
- 时间语义最强

**缺点：**
- 在当前宽度与高度下会非常拥挤
- 重叠事件和提醒事项处理复杂
- 明显超出当前下拉的设计语言

### 方案 C：顶部迷你 timeline + 底部保留列表

- 上方增加一个时间概览带
- 下方继续保留现有卡片列表

**优点：**
- 改动最小

**缺点：**
- timeline 和具体卡片关联较弱
- “现在在哪个日程里”的问题仍然没有被直接解决

## 选定方案

采用方案 A：在现有卡片列表上升级为紧凑型垂直时间线列表。

## UI/UE 设计

### 信息架构

日程区域结构调整为：

1. 日期摘要
2. timeline 日程列表
3. 全天 / 未指定时间分组

### 时间线结构

- 左侧保留约 `36pt` 的时间线窄列
- 中间是一条弱灰色主轴
- 每个时间组在主轴上有一个节点
- 节点右侧显示该时间组的卡片集合

### 节点语义

- 日历事件：实心彩色圆点，使用事件所属日历色
- 提醒事项：空心圆点，使用提醒列表颜色
- 已完成提醒：勾选态图标

### 当前时间标记

- 仅在“所选日期是今天”时显示
- 以一条红色细线横穿时间轴，并轻微延伸到右侧内容列
- 若当前落在某个事件时段内，则该时间组中的进行中事件卡片增加强调边框与左侧色条
- 若当前没有落在任何事件时段内，则当前时间线出现在下一个未来时间组之前

### 卡片视觉

保留现有卡片语言，但增加状态层级：

- 默认：浅背景卡片
- 已过去：轻微降对比
- 进行中：更高对比背景、强调边框、左缘色条
- 已选中：继续保留当前详情选中态

### 提醒事项规则

1. 有明确到期时分的提醒事项进入时间线。
2. 只有日期、没有具体时分的提醒事项进入“未指定时间”分组，不伪装成 `00:00` 项目。
3. 已过期但未完成的提醒事项仍按原到期时分出现在时间线中，不吸附到当前时间。

### 全天 / 未指定时间分组

- 全天事件和没有明确时分的提醒事项不进入 timed timeline
- 它们在时间线下方独立显示
- 不穿过“当前时间线”，避免误导

### 自动滚动

保留现有“打开今日时自动定位”的体验，但改成基于 timeline：

1. 若存在进行中项目，优先滚到该时间组
2. 否则滚到下一个未来时间组
3. 若当天已无未来项目，则滚到最后一个 timed group

## 架构设计

### 数据流

保持现有数据链路不变：

1. [`RootPopoverView`](/Users/yelog/workspace/swift/calendar-pro/CalendarPro/Views/RootPopoverView.swift) 继续负责日期选择和 EventKit 数据加载
2. [`CalendarPopoverView`](/Users/yelog/workspace/swift/calendar-pro/CalendarPro/Views/Popover/CalendarPopoverView.swift) 继续只负责页面拼装
3. [`EventListView`](/Users/yelog/workspace/swift/calendar-pro/CalendarPro/Views/Popover/EventListView.swift) 负责将 `CalendarItem` 转成 timeline 结构并渲染
4. [`EventCardView`](/Users/yelog/workspace/swift/calendar-pro/CalendarPro/Views/Popover/EventCardView.swift) 根据状态展示事件/提醒项

### 状态模型

在视图层引入临时 timeline 分组模型，用于表达：

- timed groups
- untimed groups
- 当前时间 marker 所在位置
- 某个组是否包含进行中项目
- 某个组是否整体属于过去或未来

不新增持久化字段。

### 时间刷新

当前时间线只需要分钟级刷新，不需要重新请求 EventKit。实现上在 `EventListView` 内部增加分钟级本地 timer，用于更新：

- 当前时间 marker
- 进行中状态
- 自动滚动定位条件

## 风险与处理

1. **提醒事项只有日期没有时间**
   - 风险：被错误渲染成 `00:00`
   - 处理：新增“提醒是否有明确时分”的判断逻辑

2. **同一时刻多条项目**
   - 风险：时间线左侧拥挤
   - 处理：共用一个时间节点，右侧显示多张卡片

3. **Popover 空间有限**
   - 风险：timeline 过度复杂会压缩内容可读性
   - 处理：采用紧凑列表而非完整 scheduler

4. **当前时间 marker 与点击区域冲突**
   - 风险：遮挡卡片交互
   - 处理：marker 只作视觉 overlay，不参与 hit testing

## 验证方式

1. 打开今天，确认 timeline 显示当前时间线并定位到合理位置。
2. 打开非今天日期，确认不显示当前时间线。
3. 验证事件、带时间提醒、无时间提醒、全天事件都落在正确分组。
4. 点击事件和提醒，确认原有详情交互不回退。
5. 构建并运行相关测试，确认没有编译或逻辑回归。
